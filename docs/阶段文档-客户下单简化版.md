# 五金销售管理系统阶段文档 - 客户下单简化版

## 1. 阶段目标

为解决当前系统“卖货场景不够明显”的问题，本阶段新增客户视角的小程序下单能力，让老师在演示时能够直观看到“客户注册登录 -> 浏览商品 -> 提交预定 -> 后台生成未结算销售单”的业务闭环。

本阶段坚持“尽可能简单”的原则，不新增数据库级购物车表与购物车接口，不新增独立客户资料表，尽量复用现有登录体系、销售单体系和库存扣减逻辑；前端仅增加一个业务意义上的本地临时清单，用来支持一次提交多个商品明细。

## 2. 本阶段核心结论

### 2.1 采用方案

- 新增 `CUSTOMER` 角色。
- 客户与供应商共用微信小程序。
- 小程序注册页增加“供应商注册 / 客户注册”切换。
- 客户注册成功后，只允许在微信小程序登录和使用。
- 客户登录后可浏览商品、加入清单并统一提交预定。
- 预定成功后，后端直接生成一张 `未结算` 的销售单。
- 管理端销售管理页面可查看该销售单，并继续完成后续结算。

### 2.2 明确不做

- 不新增数据库级购物车表、购物车接口、购物车持久化明细。
- 不新增独立 `customer` 客户表。
- 不接入支付功能。
- 不实现真实微信 `openid` 登录闭环，本阶段仍可沿用当前账号密码登录方式。

## 3. 为什么不做复杂购物车

当前系统销售单本身已经支持一张单包含多个商品明细，后台“新建销售单”已经具备多商品录入能力。若在小程序端继续增加完整购物车，会带来购物车表、购物车接口、购物车页面、勾选提交、数量维护、购物车转销售单等额外开发内容，但对“系统体现卖货流程”这一演示目标帮助有限。

因此最终落地采用更轻量的方式：

- 客户在商品列表页直接“加入清单”并调整数量。
- 商品详情页主要用于查看商品介绍、补充整单备注、继续加入清单。
- 清单只保存在小程序本地缓存中，不落数据库、不新增后端购物车接口。
- 客户点击“提交预定”后，前端把清单中的多个商品一次性提交到 `/api/sales/miniapp/preorder`。
- 提交后直接落为销售单，而不是先落数据库购物车。

这样既能体现客户下单，也能支持“一张销售单多个商品明细”，同时最大限度复用现有销售单逻辑。

## 4. 数据模型调整方案

## 4.1 用户表 `sys_user`

现有 `sys_user` 已具备账号、密码、昵称、手机号、openid、角色等基础字段，能够满足客户登录身份的最小需求，因此本阶段不新增客户业务表，只扩展角色枚举。

角色调整为：

- `ADMIN`：业务管理员
- `SUPPLIER`：供应商
- `CUSTOMER`：客户
- 空字符串：待申请供应商用户

说明：

- 供应商仍然保留“先注册空角色账号，再补充供应商资料并等待审核”的流程。
- 客户注册完成后直接赋予 `CUSTOMER` 角色，不需要审核。

## 4.2 销售单表 `sales_order`

建议在现有销售单表基础上新增字段：

- `customer_user_id`：关联客户用户 ID
- `order_source`：订单来源，建议值为 `MANUAL` / `MINIAPP`

保留原有字段：

- `customer_name`
- `customer_phone`

原因：

- `customer_user_id` 用于标识这张销售单属于哪个客户账号，便于客户查看“我的预定”。
- `customer_name`、`customer_phone` 继续作为下单时的快照字段保留，避免后续用户修改昵称或手机号后影响历史单据展示。
- `order_source` 用于区分“后台手工创建销售单”和“小程序客户预定销售单”，便于演示和后续统计。

## 5. 角色与端能力划分

## 5.1 管理端 Web

管理员或业务人员可执行：

- 维护商品
- 查看销售单
- 手工创建销售单
- 结算销售单
- 查看由客户小程序提交产生的未结算销售单

本阶段建议在后台销售单创建页面增加“客户选择”能力：

- 创建销售单时可下拉选择 `CUSTOMER` 用户
- 选择客户后自动带出客户昵称、手机号
- 也允许管理员手工修改显示名称或联系电话作为订单快照

## 5.2 微信小程序

### 供应商端

沿用现有能力：

- 注册
- 登录
- 提交供应商资料
- 维护报价
- 查看消息

### 客户端

新增能力：

- 注册
- 登录
- 浏览商品
- 立即预定
- 查看我的预定
- 我的信息

限制：

- 客户只能在微信小程序登录，不进入 Web 管理端。
- 客户不具备商品维护、销售结算、库存管理等后台能力。

## 6. 小程序注册与登录调整

## 6.1 注册页调整

小程序登录 / 注册页增加身份切换按钮：

- `供应商注册`
- `客户注册`

交互建议：

- 默认展示“客户注册”，因为客户流程更直接，也更贴近“卖货场景”。
- 当选择“供应商注册”时，注册成功后创建空角色账号，后续进入供应商资料完善流程。
- 当选择“客户注册”时，注册成功后直接创建 `CUSTOMER` 角色账号。

## 6.2 登录页调整

小程序登录后按角色跳转：

- `SUPPLIER`：进入供应商功能首页
- 空角色：进入供应商资料页，引导完成供应商申请
- `CUSTOMER`：进入客户商品列表页

同时限制：

- Web 后台登录仍只允许后台管理员角色进入
- 客户账号不可进入 Web 后台

## 7. 客户下单业务流程

## 7.1 注册登录流程

1. 客户进入小程序注册页。
2. 选择“客户注册”。
3. 输入用户名、密码、昵称、手机号完成注册。
4. 系统创建 `CUSTOMER` 用户并直接建立登录态。
5. 登录后进入商品列表页。

## 7.2 预定下单流程

1. 客户在小程序查看商品列表。
2. 选择某个商品进入详情，或直接在列表中点击“立即预定”。
3. 在列表页或详情页调整数量，加入清单。
4. 在列表页底部统一填写整单备注并提交预定。
5. 后端创建一张销售单，默认状态为 `未结算`。
6. 销售单可以包含多个商品明细。
7. 销售单生成后同步扣减库存，并生成收入记录，支付状态仍为 `未结算`。
8. 客户可在“我的预定”页面查看自己的订单状态。
9. 管理员可在 Web 销售单页面看到该订单，并在后续点击结算。

## 7.3 后台处理流程

1. 管理员进入销售管理列表。
2. 看到来源为“小程序客户预定”的未结算销售单。
3. 查看客户、商品、数量、金额等信息。
4. 线下完成收款后，在系统中点击“结算”。
5. 销售单和财务记录状态同步更新为 `已结算`。

## 9. 接口改造建议

本阶段目标是“接口尽量少改”。

## 9.1 认证接口

### 小程序注册接口

现有：

- `/api/auth/miniapp/register`

建议扩展请求参数：

- `registerType`：`SUPPLIER` / `CUSTOMER`

处理逻辑：

- `SUPPLIER`：创建空角色账号
- `CUSTOMER`：直接创建 `CUSTOMER` 角色账号

### 小程序登录接口

现有：

- `/api/auth/miniapp/login`

建议放行角色：

- `SUPPLIER`
- `CUSTOMER`
- 空角色

登录成功后根据角色返回前端做页面跳转。

## 9.2 商品接口

客户端只需要商品查询能力，可复用现有商品查询接口：

- `/api/product/page`
- `/api/product/{id}`

要求：

- 小程序客户只读，不允许修改商品。

## 9.3 客户预定接口

建议新增接口：

- `/api/sales/miniapp/preorder`

作用：

- 当前登录客户提交预定信息后，直接创建销售单
- 支持一次提交多个商品明细

建议请求体：

```json
{
  "items": [
    {
      "productId": 1,
      "quantity": 2
    }
  ],
  "remark": "到店自提"
}
```

后端处理建议：

- 根据当前登录用户自动识别 `customer_user_id`
- 自动写入 `customer_name`、`customer_phone`
- 单价默认取商品当前 `sale_price`
- 自动计算总金额
- 自动生成 `payment_status = 0` 的销售单
- 自动写入 `order_source = MINIAPP`

## 9.4 我的预定接口

建议新增接口：

- `/api/sales/miniapp/my-page`
- `/api/sales/miniapp/{id}`

作用：

- 查询当前客户自己的预定记录
- 查看当前客户某张预定单详情

权限要求：

- 只能查看当前登录客户自己的订单

## 9.5 管理端客户下拉接口

建议新增或复用用户查询接口，支持按角色查询：

- `/api/system/user/page?role=CUSTOMER`

作用：

- 后台销售单创建页下拉选择客户

## 10. 权限控制建议

## 10.1 小程序端

- `CUSTOMER` 只能访问客户商品、客户订单、个人中心相关页面
- `SUPPLIER` 只能访问供应商资料、报价、消息相关页面
- 空角色用户只能继续完善供应商申请资料

## 10.2 后端接口

- 客户预定接口仅允许 `CUSTOMER` 调用
- 客户订单查询接口仅允许 `CUSTOMER` 查询自己的订单
- 后台销售管理接口仅允许后台管理角色访问
- 商品维护、库存、财务、用户管理等接口不对客户开放

## 11. 演示效果说明

本阶段完成后，系统演示路径将更加突出“卖货”主题：

1. 客户在小程序完成注册登录。
2. 客户浏览商品并立即预定。
3. 系统自动生成未结算销售单。
4. 管理员在 Web 端看到客户订单。
5. 管理员完成线下收款后点击结算。

这样能够非常直观地展示：

- 有客户角色
- 有客户下单行为
- 有销售单生成
- 有未结算到已结算的销售流程

相比原先仅由后台手工录入销售单，这一版本更容易体现“这是一个卖货系统”。

## 12. 分阶段实施建议

## 第一阶段：最小可演示版本

- `sys_user.role` 增加 `CUSTOMER`
- 注册页增加“供应商注册 / 客户注册”切换
- 小程序客户登录成功后进入商品列表
- 新增客户立即预定接口
- 生成未结算销售单
- 后台销售列表可看到客户订单

## 第二阶段：体验补充

- 增加“我的预定”页面
- 销售单增加来源字段展示
- 后台创建销售单支持客户下拉选择

## 第三阶段：可选增强

- 增加订单取消能力
- 增加客户收货地址字段
- 如后续业务复杂度上升，再考虑数据库级购物车

## 13. 最终实施原则

本阶段以“少改动、能演示、突出卖货场景”为最高优先级，最终实施原则如下：

- 不新增 `customer` 表
- 优先复用 `sys_user`
- 不做数据库级购物车
- 不做支付
- 先做轻量本地清单聚合，再统一提交预定
- 先打通销售单闭环，再考虑体验优化

该方案能够以较低开发成本，显著增强系统的销售业务展示效果，适合作为当前毕设阶段版本推进。
